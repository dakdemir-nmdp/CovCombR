Overview

This vignette provides a deep dive into CovCombR’s advanced configuration options. It is intended for users who need fine-grained control over the estimation process or want to understand the mathematical details behind the dual-scale parameterization.

Topics covered:

  1. Understanding output scales (Σ̂ vs Ŝ)
  2. Scale parameter estimation (α parameters)
  3. Initialization strategies
  4. Control parameters for convergence and stability
  5. Bootstrap configuration
  6. SEM (Supplemented EM) configuration

Prerequisites: Familiarity with basic usage of fit_covcomb(). See vignette("CovCombR-vignette") for an introduction.

1. Understanding Output Scales: Σ̂ vs Ŝ

Why Two Outputs?

fit_covcomb() returns two covariance matrices:

  • $Sigma_hat (accessed via coef(fit)): The primary output
  • $S_hat (accessed via fitted(fit)): An alternative scale presentation

TL;DR for most users: Use Sigma_hat for everything. Ignore S_hat unless you have a specific reason to use it.

Mathematical Relationship

The model assumes each study k observes:

\[\mathbf{S}_k \sim \text{Wishart}(\nu_k, \alpha_k \boldsymbol{\Sigma})\]

where: - \(\boldsymbol{\Sigma}\) = common underlying covariance (what we want to estimate) - \(\alpha_k\) = study-specific scale factor

When scale_method = "none" (default): - All \(\alpha_k = 1\) (fixed) - \(\boldsymbol{\Sigma}\) estimated directly - Result: Sigma_hat = S_hat (identical)

When scale_method = "estimate": - \(\alpha_k\) estimated jointly with \(\boldsymbol{\Sigma}\) - Two normalization choices: geometric mean (default) or arithmetic mean - Result: Sigma_hat and S_hat may differ

Normalization Schemes

Geometric mean normalization (default, alpha_normalization = "geometric"):

\[\prod_{k=1}^K \alpha_k = 1 \quad \Rightarrow \quad \text{Sigma\_hat} = \text{S\_hat}\]

The α parameters are normalized so their geometric mean is 1. This makes Σ̂ and Ŝ identical.

Arithmetic mean normalization (alpha_normalization = "arithmetic"):

\[\frac{1}{K} \sum_{k=1}^K \alpha_k = 1\]

The α parameters are normalized so their arithmetic mean is 1. In this case:

\[\text{S\_hat} = \left(\frac{1}{K} \sum_{k=1}^K \alpha_k\right) \times \text{Sigma\_hat}\]

Which to use? - For reporting results: Always use Sigma_hat (via coef(fit)) - For method comparison: S_hat can provide an alternative view when using arithmetic normalization

Example: Comparing Normalizations

library(CovCombR)

# Simulate data with heterogeneous scales
set.seed(2025)
p <- 6
var_names <- paste0("V", 1:p)
true_Sigma <- diag(p)
dimnames(true_Sigma) <- list(var_names, var_names)

# Study 1: standard scale (α = 1)
S1 <- rWishart(1, 50, true_Sigma[1:4, 1:4])[,,1] / 50
dimnames(S1) <- list(var_names[1:4], var_names[1:4])

# Study 2: inflated scale (α = 2)
S2 <- rWishart(1, 60, 2 * true_Sigma[3:6, 3:6])[,,1] / 60
dimnames(S2) <- list(var_names[3:6], var_names[3:6])

# Fit with geometric normalization (default)
fit_geom <- fit_covcomb(
  S_list = list(S1 = S1, S2 = S2),
  nu = c(S1 = 50, S2 = 60),
  scale_method = "estimate",
  alpha_normalization = "geometric"
)

# Fit with arithmetic normalization
fit_arith <- fit_covcomb(
  S_list = list(S1 = S1, S2 = S2),
  nu = c(S1 = 50, S2 = 60),
  scale_method = "estimate",
  alpha_normalization = "arithmetic"
)

# Compare
cat("Geometric normalization:\n")
#> Geometric normalization:
cat("  Alpha:", fit_geom$alpha_hat, "\n")
#>   Alpha: 0.7416765 1.348297
cat("  Sigma_hat[1,1]:", coef(fit_geom)[1,1], "\n")
#>   Sigma_hat[1,1]: 1.488499
cat("  S_hat[1,1]:", fitted(fit_geom)[1,1], "\n")
#>   S_hat[1,1]: 1.488499
cat("  Identical?", all.equal(coef(fit_geom), fitted(fit_geom)), "\n\n")
#>   Identical? TRUE

cat("Arithmetic normalization:\n")
#> Arithmetic normalization:
cat("  Alpha:", fit_arith$alpha_hat, "\n")
#>   Alpha: 0.7097426 1.290257
cat("  Sigma_hat[1,1]:", coef(fit_arith)[1,1], "\n")
#>   Sigma_hat[1,1]: 1.555463
cat("  S_hat[1,1]:", fitted(fit_arith)[1,1], "\n")
#>   S_hat[1,1]: 1.555463
cat("  Identical?", all.equal(coef(fit_arith), fitted(fit_arith)), "\n")
#>   Identical? TRUE

Key takeaway: Use geometric normalization (default) to avoid confusion between Σ̂ and Ŝ.

2. Scale Parameter Estimation

When to Use scale_method = "estimate"

Estimate scale factors (\(\alpha_k\)) when:

✓ Studies use different platforms or technologies - Example: Combining GRMs from SNP array (100K markers) and whole-genome sequencing (10M markers) - Scale factors account for different marker densities

✓ Different measurement units - Example: Combining correlation matrices (scale ≈ 1) with covariance matrices (scale variable)

✓ Suspected heterogeneous variance - Studies measure the same variables but in populations with different variability

✗ Do NOT use when: - All studies use identical protocols - You’ve already standardized the matrices - Small sample sizes (α estimation requires data; use scale_method = "none" with fewer than ~50 samples per study)

Interpreting Alpha Parameters

# Use the example data from setup
fit <- fit_covcomb(S_list, nu, scale_method = "estimate")
cat("Alpha parameters:\n")
#> Alpha parameters:
print(fit$alpha_hat)
#>    Study1    Study2 
#> 1.0489011 0.9533787

Interpretation: - α_k > 1: Study k has larger variance than average - α_k < 1: Study k has smaller variance than average - α_k ≈ 1: Study k is at the average scale

Example: - Study 1 (α = 0.85): 15% below average variance - Study 2 (α = 1.20): 20% above average variance - Study 3 (α = 0.95): 5% below average variance

Scale Estimation with GRMs

Genomic relationship matrices (GRMs) from different marker panels can have different scales due to differing numbers of markers. Scale estimation accounts for this.

# Example workflow (hypothetical - requires actual genotype data)
# GRM from 50K SNP array
GRM_50K <- compute_grm(genotypes_50K)  # Your GRM computation function

# GRM from 500K SNP array
GRM_500K <- compute_grm(genotypes_500K)

# Combine with scale estimation
fit <- fit_covcomb(
  S_list = list(Array50K = GRM_50K, Array500K = GRM_500K),
  nu = c(Array50K = 50000, Array500K = 500000),
  scale_method = "estimate"
)

# Check relative scales
print(fit$alpha_hat)
# Expected: alpha for 500K might be larger due to finer resolution

3. Initialization Strategies

Available Initialization Methods

The init_sigma parameter controls how the algorithm starts:

1. "avg_padded" (default): - Compute study-wise average covariance for observed blocks - Pad missing blocks with small values (ridge parameter) - Pro: Warm start, faster convergence - Con: Can get stuck in local optima if initial guess is poor

2. "identity": - Start with identity matrix - Pro: Unbiased start, good for difficult problems - Con: Slower convergence (more iterations needed)

3. Custom matrix: - Provide your own starting value - Use case: Re-fit with previous estimate, or use domain knowledge

When to Change Initialization

Use "identity" if: - Algorithm converges to unexpected values - Suspecting multi-modality (multiple local optima) - High condition number warnings

Use custom init if: - Fitting many similar datasets (use previous fit as init) - Have strong prior information about the covariance structure

Example: Comparing Initializations

# Difficult case: chain topology with small p
set.seed(2025)
var_names <- paste0("V", 1:5)
true_Sigma <- diag(5)
dimnames(true_Sigma) <- list(var_names, var_names)

S1 <- rWishart(1, 100, true_Sigma[1:3, 1:3])[,,1] / 100
dimnames(S1) <- list(var_names[1:3], var_names[1:3])

S2 <- rWishart(1, 100, true_Sigma[3:5, 3:5])[,,1] / 100
dimnames(S2) <- list(var_names[3:5], var_names[3:5])

# Default initialization
fit_default <- fit_covcomb(
  S_list = list(S1 = S1, S2 = S2),
  nu = c(S1 = 100, S2 = 100),
  init_sigma = "avg_padded"
)

# Identity initialization
fit_identity <- fit_covcomb(
  S_list = list(S1 = S1, S2 = S2),
  nu = c(S1 = 100, S2 = 100),
  init_sigma = "identity"
)

# Compare convergence
cat("Avg padded iterations:", fit_default$convergence$iterations, "\n")
#> Avg padded iterations: 500
cat("Identity iterations:", fit_identity$convergence$iterations, "\n")
#> Identity iterations: 500

# Compare estimates
cat("Max difference:", max(abs(coef(fit_default) - coef(fit_identity))), "\n")
#> Max difference: 0.1975073

4. Control Parameters

Convergence Control

max_iter (default: 1000): - Maximum number of EM iterations - Increase if algorithm doesn’t converge - Typical range: 50-2000

tol (default: 1e-6): - Relative change threshold for convergence - Algorithm stops when \(\|\Sigma^{(t+1)} - \Sigma^{(t)}\|_F / \|\Sigma^{(t)}\|_F <\) tol - Looser tolerance (1e-4): Faster, less accurate - Tighter tolerance (1e-8): Slower, more accurate

# Quick and dirty fit
fit_fast <- fit_covcomb(S_list, nu,
                        control = list(max_iter = 100, tol = 1e-4))

# High precision fit
fit_precise <- fit_covcomb(S_list, nu,
                           control = list(max_iter = 2000, tol = 1e-8))

Numerical Stability Control

ridge (default: 1e-8): - Added to diagonal during matrix inversions - Prevents singularity when matrices are nearly singular - Increase (1e-6 or 1e-5) if: - Getting “matrix is singular” errors - Very small eigenvalues in estimates - Weak identifiability warnings - Decrease (1e-10) for high-precision problems with strong identifiability

min_eigen (default: 1e-6): - Minimum eigenvalue enforced during positive definite projection - Eigenvalues below this threshold are set to min_eigen - Larger values (1e-4): More aggressive PD enforcement - Smaller values (1e-8): Preserve near-zero eigenvalues

# Problematic dataset with near-singularity
fit_stable <- fit_covcomb(S_list, nu,
                          control = list(ridge = 1e-5, min_eigen = 1e-4))

Control Parameter Interaction

General rules: - If convergence is slow → relax tol, increase max_iter - If numerical instability → increase ridge and min_eigen - If unexpected estimates → try init_sigma = "identity"

5. Bootstrap Configuration

Key Parameters

When using se_method = "bootstrap", configure via control$bootstrap:

B (default: 100): - Number of bootstrap replicates - Recommendation: - Exploratory: B = 50 - Publication: B = 200 - High stakes: B = 500+ - Trade-off: Larger B = more accurate SEs, longer runtime

seed (default: NULL): - Set for reproducible results - Always set in production code!

parallel (default: FALSE): - Use parallel computation (via parallel package) - Caution: Currently experimental - Set cores to number of CPU cores (default: detectCores() - 1)

retain_samples (default: FALSE): - Store all bootstrap estimates - Useful for distribution analysis, quantile-based inference - Warning: High memory usage for large p

Example: Publication-Quality Bootstrap

# Note: In practice use B=200 for publication; here we show workflow
fit_pub <- fit_covcomb(
  S_list = S_list,
  nu = nu,
  se_method = "bootstrap",
  control = list(
    bootstrap = list(
      B = 200,              # 200 replicates for publication
      seed = 2025,          # Reproducibility
      parallel = FALSE,     # Serial execution (safer)
      retain_samples = TRUE # Keep all bootstrap estimates
    )
  )
)

# Access bootstrap samples
boot_samples <- fit_pub$bootstrap_samples  # Array: p x p x B

# Compute 95% confidence intervals (quantile method)
Sigma_hat <- coef(fit_pub)
CI_lower <- apply(boot_samples, 1:2, quantile, probs = 0.025)
CI_upper <- apply(boot_samples, 1:2, quantile, probs = 0.975)

# Hypothesis test: Is Sigma[1,2] significantly different from 0?
cat("Estimate:", Sigma_hat[1,2], "\n")
cat("95% CI: [", CI_lower[1,2], ",", CI_upper[1,2], "]\n")

Monitoring Bootstrap Convergence

Check if individual bootstrap fits converge:

# Bootstrap convergence monitoring (use smaller B for speed)
fit <- fit_covcomb(S_list, nu, se_method = "bootstrap",
                  control = list(bootstrap = list(B = 200, seed = 2025)))

6. SEM (Supplemented EM) Configuration

When to Use SEM

Supplemented EM (Meng & Rubin, 1991) provides standard errors based on the observed information matrix without bootstrap resampling.

Use SEM if: - Large p (> 100) where bootstrap is prohibitively slow - Need fast approximate SEs for diagnostics - Condition: Well-identified problem (good overlap, not too sparse)

Do NOT use SEM if: - Small/moderate p (< 50) → use bootstrap instead - Weak identifiability (high condition number) - Safety-critical application → bootstrap is gold standard

SEM Control Parameters

Configure via control$sem:

h (default: 1e-6): - Finite difference step size for numerical differentiation - Smaller h = more accurate but less stable - Typical range: 1e-8 to 1e-5

ridge (default: inherits from control$ridge): - Ridge parameter for rate matrix inversion - Increase if rate matrix is singular

Example: SEM with Validation

# Large problem where bootstrap is slow (hypothetical workflow)
set.seed(2025)
p <- 50
var_names <- paste0("V", 1:p)

# Create large study data (simulation code omitted for brevity)
# S_list_large <- create_large_study_data(p = 50, K = 3)
# nu_large <- c(Study1 = 500, Study2 = 600, Study3 = 550)

# Fit with SEM
fit_sem <- fit_covcomb(
  S_list = S_list_large,
  nu = nu_large,
  se_method = "sem",
  control = list(
    sem = list(h = 1e-6, ridge = 1e-6)
  )
)

# CRITICAL: Validate SEM against bootstrap on a subset
p_subset <- 10
indices <- 1:p_subset

S_list_subset <- lapply(S_list_large, function(S) {
  S[indices, indices]
})

fit_boot_subset <- fit_covcomb(
  S_list = S_list_subset,
  nu = nu_large,
  se_method = "bootstrap",
  control = list(bootstrap = list(B = 200, seed = 2025))
)

fit_sem_subset <- fit_covcomb(
  S_list = S_list_subset,
  nu = nu_large,
  se_method = "sem",
  control = list(sem = list(h = 1e-6))
)

# Compare SEs
se_boot <- fit_boot_subset$Sigma_se
se_sem <- fit_sem_subset$Sigma_se

cat("Mean SE (bootstrap):", mean(se_boot, na.rm = TRUE), "\n")
cat("Mean SE (SEM):", mean(se_sem, na.rm = TRUE), "\n")
cat("Correlation:", cor(as.vector(se_boot), as.vector(se_sem),
                        use = "complete.obs"), "\n")

SEM Diagnostics

Check condition number of rate matrix: - SEM computes the EM rate matrix (derivative of EM operator) - If this matrix is ill-conditioned, SEs are unreliable - Warnings will be issued automatically

Fallback strategy: If SEM produces warnings or unrealistic SEs, fall back to bootstrap for final analysis.

Summary: Configuration Decision Tree

Do you need standard errors?
├─ NO → se_method = "none", use defaults
└─ YES
   ├─ Small problem (p < 50)?
   │  └─ YES → se_method = "bootstrap", B = 200
   └─ Large problem (p > 50)?
      └─ YES → se_method = "sem", validate on subset

Are studies from different platforms/scales?
├─ YES → scale_method = "estimate"
└─ NO → scale_method = "none" (default)

Having convergence issues?
├─ Slow convergence → increase max_iter, relax tol
├─ Unexpected values → try init_sigma = "identity"
└─ Singularity errors → increase ridge and min_eigen

Need reproducible results?
└─ YES → set control$bootstrap$seed
